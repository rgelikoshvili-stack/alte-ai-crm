## Observed Issue

The Netlify Pro v2 chatbot UI showed broken Georgian/mojibake text in chat bubbles and UI labels:

```text
ГЎЖ’...
```

The reported visible impact included:

- Georgian user-facing chat text appeared broken.
- Department/operator/source-adjacent labels could display mojibake.
- Georgian UI messages generated by widget actions were not readable.

## Root Cause

Backend raw response checks show production JSON preserves Georgian Unicode and does not contain the mojibake marker.

The root cause was frontend source corruption in active Pro v2 widget files. `pro-v2-chat.jsx` contained hardcoded Georgian UI strings already saved as mojibake, for example operator labels, handover keywords, copied/toast text, attachment labels, and contact error copy.

Netlify also served `.jsx` variant files as `application/octet-stream`, so the fix adds an explicit UTF-8 content type for `/variants/*.jsx` to remove runtime decoding ambiguity for Babel-loaded variant files.

## Fix Summary

- Replaced mojibake Georgian literals in the active Pro v2 chat variant with valid UTF-8 Georgian strings.
- Preserved backend endpoint usage and no-contact behavior.
- Added Netlify UTF-8 delivery header for JSX variant files:

```toml
[[headers]]
  for = "/variants/*.jsx"
  [headers.values]
    Content-Type = "text/plain; charset=UTF-8"
```

- Added production smoke coverage for raw Georgian API encoding.
- Added visual QA coverage that sends a Georgian test question and asserts visible UI text does not contain `ГЎЖ’` and does contain Georgian script.
